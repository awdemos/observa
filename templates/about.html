About Observa

What every number means, where the data comes from, and how to read the dashboard.

Metrics Glossary

CPU usage
Percentage of total CPU time currently in use across all cores. Sampled from sysinfo on the host. Higher values mean more load.
Per-core usage
Individual CPU core utilization percentages. Useful for spotting single-core saturation or uneven load distribution.
CPU frequency
Current clock speed in MHz. May fluctuate with power-saving or boost states.
Memory used / free / total
Physical RAM statistics. Used is what applications and caches are consuming; free is unused; total is installed RAM.
Memory %
used / total × 100. High values do not always indicate a problem because modern systems keep data in caches, but sustained growth can signal a leak.
Swap
Virtual memory backed by disk. Used when physical RAM is exhausted. Heavy swap usage usually degrades performance.
Disk usage
Percentage of each block device's capacity that is allocated. Independent of read/write activity.
Disk read / write rate
Current throughput for each disk in bytes per second. Spikes correlate with I/O-bound workloads.
Network RX / TX
RX (receive) and TX (transmit) cumulative byte counters since the interface came up. The dashboard also shows current rates in bytes per second.
Process CPU / memory
Per-process CPU percentage and resident memory. CPU is relative to one core; a process can exceed 100% on multi-core systems.
GPU usage / memory
GPU compute utilization and VRAM usage when a compatible GPU is present and detectable.
AI Servers
Detected local inference processes (llama.cpp, vLLM, Ollama, Triton, etc.) identified by process name and optional port hint.
Severity
Log/alert classification: Debug, Info, Warn, Error, Critical.

Where the Data Comes From

System metrics
Collected by the observa-collector crate using sysinfo. Sampling interval is configurable (default 2000 ms).
Logs
Ingested from journalctl when available, or from a configured fallback log file. Parsed into structured events by observa-ingestor.
Persistence
SQLite stores metric history, logs, and chat sessions. Retention and optional zstd compression are configurable.
Cache
Optional Redis layer speeds up recent lookups. If Redis is unavailable the server falls back to in-memory caching.
Live updates
The /events endpoint streams Server-Sent Events (SSE) to the browser so cards and tables update without a full page reload.

How to Read the Dashboard

  • Live metrics update automatically via SSE when the status pill shows "live".
  • High CPU / memory / disk % usually warrants investigation.
  • High RX/TX rates are not inherently bad; compare them to the baseline for the workload.
  • Swap activity often indicates memory pressure.
  • Critical / Error logs are surfaced on the Security page.
  • Every chart and card has an expandable Raw data section so you can verify the exact numbers.
